### 总体架构设计方案：流水线 (Pipeline) 模式

我们不再让用户在 `main.py` 里手动切来切去（暴力模式 or 规则模式），而是将 `main.py` 升级为一个**自动化流水线调度器**。

#### 1. 预处理校验机制
*   在 `main.py` 接收到 `input_file` 时，强制检查路径是否包含 `processed_story`（或者是否是通过预处理模块生成的）。
*   如果用户直接传入了其他路径，通过 `logger` 打印黄色的警告信息，提醒用户：“您当前处理的似乎不是预处理后的标准文本，建议先经过预处理再执行纠错流水线！”

#### 2. 流水线阶段设计

整个纠错过程将分为四个阶段，依次执行，上一个阶段的输出直接作为下一个阶段的输入：
# 
*   **Stage 1：全局清道夫 (对应 `fix_story_v1`)**
    *   **动作**：调用 `corrector_scan.py` (暴力扫描)。
    *   **输入**：`data/processed_story/第X卷.txt` （用户指定）
    *   **输出**：`output/fix_story_v1/第X卷.txt` （可以在src\config\settings.py进行修改，默认是fix_story_v1）
    *   **目的**：大面积修复OCR错误，并将异形包裹符号（如 `""`、`『』`）批量规整为「」。

*   **Stage 2：第一轮交替规则收敛 (对应 `fix_story_v2`)**
    *   **动作**：调用 `corrector_rule.py` (规则扫描，寻找「」连续异常)。
    *   **输入**：`output/fix_story_v1/第X卷.txt` 
    *   **输出**：`output/fix_story_v2/第X卷.txt` （可以在setting.py进行修改，默认是fix_story_v2）
    *   **目的**：修复 V1 遗漏或新产生的「」不交替问题。

*   **Stage 3：中括号 [ ] 异常清缴 (对应 `fix_story_v3`)**
    *   **动作**：复用并改造 `corrector_rule.py` 的逻辑（可以通过增加参数或新建 `corrector_bracket.py`）。这个新逻辑将专门扫描文本中的 `[` 和 `]`，一旦发现，就以其为中心提取上下文切块交给大模型。
    *   **输入**：`output/fix_story_v2/第X卷.txt`
    *   **输出**：`output/fix_story_v3/第X卷.txt` （可以在setting.py进行修改，默认是fix_story_v3）
    *   **目的**：解决你提到的那种“错误符号恰好满足交替规则而被遗漏”的边界情况。

*   **Stage 4：最终交替规则兜底 (对应 `fix_story_v4`)**
    *   **动作**：再次调用 `corrector_rule.py`。
    *   **输入**：`output/fix_story_v3/第X卷.txt`
    *   **输出**：`output/fix_story_v4/第X卷.txt` （可以在setting.py进行修改，默认是fix_story_v4）
    *   **目的**：防止 V3 在修改 `[]` 时破坏了原有的「」交替平衡，作为最后一道防线。

### 具体代码修改规划：

1.  **扩展 `src/modules/utils.py`**：
    *   修改 `find_consecutive_symbols` 函数，或者新增一个 `find_specific_symbols` 函数，让它能够不只寻找连续的符号，还能寻找指定的**独立异常符号**（如 `[` 和 `]`），返回其索引。

2.  **重构或新增规则引擎**：
    *   方案 A（推荐）：将 `corrector_rule.py` 升级，增加一个 `target_symbols` 参数。默认找连续的「」，如果传了 `[`，就去定位并修正 `[`。
    *   方案 B：新建一个 `corrector_bracket.py`，专门复制 `corrector_rule.py` 的代码用来处理 `[]`。（方案 A 更优雅，代码复用率高，但我会根据实际情况看哪种更好实现）。
    选择方案B，但是可以从corrector_rule.py中导入需要的函数，而不是无脑全部复制然后再修改。该流程特定的内容无法复用的，才需要自己写。

3.  **改造 `main.py` 的调度逻辑**：
    *   移除原先的 `--full_scan` 互斥选项。
    *   新增一个 `--pipeline` 参数。如果开启，程序自动按照 `V1 -> V2 -> V3 -> V4` 的顺序连续执行。
    *   用户仍然可以通过 `--stage 1` 或 `--stage 3` 的参数，单独执行某一个步骤（方便调试）。

4.  **修改 prompt (可能需要)**：
    *   在处理 `[]` 的 Stage 3 中，可能需要给大模型一个稍微不同的 prompt，明确告诉它“注意修复文本中不合规的 `[]` 符号”。
