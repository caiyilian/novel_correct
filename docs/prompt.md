## 一、项目基础信息与核心目标
### 1.1 待处理文件基础信息
- 待处理文件：`ori_story`文件夹下共10卷小说文本，文件命名为「第1卷.txt」至「第10卷.txt」
- 文件格式规范：换行符为Unix (LF)格式，字符编码为UTF-16
- 文本核心特征：小说包含旁白与人物对话，标准人物对话以日文引号「（开符号）、」（闭符号）成对包裹；文本存在OCR识别误差、翻译录入失误导致的符号错误、错别字、格式混乱、符号顺序异常等问题。

### 1.2 核心项目目标
1.  **核心纠错目标**：自动化修正小说中所有对话符号错误，将所有人物对话的包裹符号统一规范为成对的「」，修正符号混用、漏写、错写、顺序颠倒、开闭不匹配等全场景错误。
2.  **全链路可追溯目标**：对所有修正行为进行100%可追溯的记录，精准定位每个修改点在原始文本中的绝对位置、原内容、修改后内容，支持人工复核与二次校准。
3.  **防幻觉与校验目标**：通过多层级校验机制，规避大模型幻觉导致的原文篡改，保障修正后的文本无符号错误、无顺序错误、无漏改误改，最大化降低修正失误率。
4.  **兼容处理目标**：兼容修正OCR识别导致的明显错别字与乱码，同时严格区分「有效错别字修正」与「大模型幻觉篡改」，避免误判与误操作。

### 1.3 核心设计原则
- **准确率优先原则**：不追求处理速度，只追求修正准确率，允许无上限次数的大模型调用，允许单卷文本多轮迭代修正，直至通过所有校验规则。
- **防幻觉原则**：所有设计优先规避大模型幻觉，包括上下文长度控制、输出格式强约束、修改后二次复核机制。

## 二、对话符号规范与纠错边界定义
### 2.1 正确的对话格式规范
小说中所有人物的口语对话，必须以**成对的「（开符号）和」（闭符号）** 进行包裹，开符号与闭符号必须一一对应、交替出现，不得出现嵌套、连续重复、漏写、错写。
- 正确示例：`「请问叶勒在哪里啊?」`
- 其他任何符号包裹的人物对话，均属于错误格式，需按本规范统一修正。

### 2.2 必须修正的错误场景
1.  **符号混用错误**：人物对话使用非「」的符号作为开/闭符号，包括但不限于`[ ]`、`“ ”`、`『 』`、英文双引号`""`、英文单引号`''`等，需将对话包裹符号统一替换为「」。
2.  **开闭符号不匹配错误**：对话开符号与闭符号不对应，例如开符号用`[`、闭符号用`」`，开符号用`「`、闭符号用`”`等，需统一修正为成对的「」。
3.  **符号缺失错误**：对话漏写开符号「或漏写闭符号」，导致成对数量不匹配、顺序断裂，需结合上下文补全正确的符号。
4.  **符号顺序错误**：开闭符号顺序颠倒，例如连续出现两个「、连续出现两个」，或闭符号出现在开符号之前；即使总数量相等，也需结合上下文修正为正确的交替顺序。
5.  **OCR识别错误**：因OCR识别导致的符号乱码、错字替代（例如将「识别为`;`、将「识别为乱码「垣」等），需结合上下文还原为正确的「」符号与文本内容。
6.  **长对话换行错误**：单段对话因过长被手动换行，导致换行后的对话内容脱离符号包裹，需结合上下文确认对话边界，保障完整对话被成对「」包裹。

### 2.3 严格禁止修改的场景
1.  **非对话场景的符号使用**：当`『 』`仅用于包裹专有名词、特定指代内容、特殊称谓时，必须保留原符号，不得修改为「」。
    - 示例：`今年应该是他们田里的某个人会是『赫萝』吧`中，『赫萝』为特定指代，禁止修改。
2.  **原文旁白内容**：除明确的错误符号、OCR识别错别字外，不得修改原文的旁白文本、剧情内容、人物名称、专有名词，禁止大模型对原文进行润色、改写、扩写、缩写。


### 2.4 可选择性修正的补充规则
- 错别字修正：仅修正OCR识别导致的明显错别字（示例：将「声晋」修正为「声音」），不得修改原文的用词、语法、句式；所有错别字修正必须同步纳入变更记录，与符号修正区分标记。

## 三、技术栈与核心约束
### 3.1 强制技术栈要求
- 开发语言：Python
- 大模型调用：基于Ollama服务的HTTP API实现，参考提供的`test_ollama.py`示例代码开发。
- 第三方库：优先使用成熟稳定的主流库实现核心功能，例如`difflib`用于文本差异对比、`requests`用于接口调用、`re`用于正则预处理。
- 数据存储：变更记录、日志、校验结果使用JSON/CSV格式存储，保障可读取、可追溯。

### 3.2 大模型调用规范
- 固定配置信息：
  - 目标模型：`qwen2.5-coder:14b`
  - 服务地址：`http://172.31.102.189:11434`
  - API端点：Ollama的`/api/generate`接口，使用非流式响应（`stream: False`）
- 模型参数要求：`temperature`固定为0.1-0.3，最大化降低随机性与幻觉；`num_predict`根据输入文本长度动态适配，保障输出完整。
- 调用提示词强约束：每次调用大模型时，必须在prompt中明确以下强制要求，不得省略：
  1.  仅修正输入文本中人物对话的包裹符号错误与OCR识别导致的明显错别字，不得修改其他任何内容；
  2.  必须严格保留输入文本的所有换行符、段落结构、空格缩进，不得新增、删除、调整任何换行与格式；
  3.  仅返回修改后的完整文本，不得返回任何解释说明、备注、额外内容；
  4.  不得对原文进行润色、扩写、缩写、改写，不得改变原文的任何剧情与表述。
- 异常重试机制：支持超时重试、接口异常重传，重试次数可配置，避免因网络波动、服务异常导致处理中断。

### 3.3 核心设计约束
- 上下文长度约束：绝对不允许一次性把一整卷输入给大模型，虽然大模型上下文窗口大，如果要求修改的内容过多，会导致幻觉出现，可以参考example.md这个例子
- 切块边界约束：文本切块时，严禁将单段完整对话从中间切断，避免大模型因上下文缺失导致误补符号、误修正；优先采用「大上下文窗口+仅修改中间核心区域」的方案，即给大模型输入超出核心处理区域的上下文，但明确要求其仅修改中间指定的核心区域，彻底规避切块边界问题。
- 索引精准性约束：必须解决文本修改后长度变化导致的原始索引偏移问题，所有变更记录必须锚定**原始文本的绝对字符索引**，不得使用修改后文本的索引。
- 多轮迭代约束：允许单卷文本进行多轮纠错、多轮复核，直至通过所有校验规则，无需限制迭代次数。

## 四、系统核心功能模块详细需求
### 4.1 文件读写模块
1.  严格按照UTF-16编码、Unix (LF)换行符格式读取与写入文件，不得改变文件的编码与换行格式。
2.  支持单卷文件单独处理，也支持10卷文件批量处理。
3.  写入修改后的文件时，不得覆盖原始文件，需输出到指定的独立文件夹，永久保留原始文件不动。
4.  完善的异常捕获机制，处理文件不存在、编码错误、权限不足等异常，输出清晰的错误日志。

### 4.2 文本预处理模块
1.  连续换行标准化：通过正则表达式将文本中连续多个`\n`替换为1个`\n`，保留单换行、双换行的原文格式，避免无效空行干扰切块与大模型处理。使用re.sub(r'\n+', '\n', re.sub(r'[^\S\n]', '', text)) 即可完成
2.  预处理完的小说保存到processed_story文件夹中，下一步操作基于此文件夹中的文件进行。

### 4.3 智能文本切块模块
1.  切块长度可配置，单块文本长度适配模型上下文窗口，预留足够的冗余空间。
2.  上下文重叠机制：相邻两个文本块之间设置可配置的重叠上下文区域，保障对话的完整性，避免边界处的对话漏改。
3.  使用「大上下文+核心修改区」模式：可输入更大范围的上下文文本，同时明确标记仅需修改的中间核心区域，让大模型仅对核心区域进行修正，彻底规避切块边界导致的误修正问题。
4.  大模型返回的内容需要用re.sub(r'\n+', '\n', re.sub(r'[^\S\n]', '', text))进行清洗，然后再对比，只查看核心修改区域的修改内容，使用difflib进行差异对比。

### 4.4 大模型纠错执行模块
1.  基于标准化的prompt模板发起调用，每次调用均携带完整的约束规则，禁止无约束的大模型调用。
2.  **输出后处理**：在校验与比对前，必须对大模型返回的文本进行严格清洗，使用re.sub(r'\n+', '\n', re.sub(r'[^\S\n]', '', text))完成
3.  **输出校验**：对清洗后的文本进行逻辑校验。**放宽对长度变化的限制**，允许因错别字修正或符号补全导致的合理长度差异；但需重点监控结构性破坏，若检测到换行符数量急剧减少（如丢失超过20%）或出现明显的非原文内容（如大段新增解释性文字），则判定为异常输出并触发重试。并且在error文件夹中创建文件记录这个异常（包括发送给大模型的内容、prompt、大模型返回的内容），每个异常单独一个文件。
4.  支持单文本块多次重复调用，若多次调用后仍出现异常输出，标记该文本块为异常，暂停处理并输出日志，交由人工复核。
5.  大模型修改一个内容块需要一定时间，一卷会被分为大量内容块，所以处理一整卷肯定也更耗时，需要耐心等待，如果某一卷的处理时间异常的短（小于10s）那就得检查一下是不是没有成功调用大模型

### 4.5 差异比对与变更全记录模块
1.  基于逐字符级的文本差异比对，精准识别每一处修改的位置、原内容、修改后内容，区分「符号修正」「错别字修正」「符号增删」三类修改类型。

### 4.6 二次复核与修正校准模块
1.  基于变更记录，以每个修改点为中心，提取其前后足够的上下文文本，发送给大模型进行专项复核。
2.  复核prompt必须明确告知大模型原始修改内容，要求其判断修改是否正确、是否符合规则，若不正确则给出正确的修正内容。
3.  复核后的修正内容，同步更新到变更记录中，单独存储到复核专用的记录列表，不得覆盖原始修改记录，保障全链路可追溯。
4.  对于校验模块识别出的符号数量不匹配、顺序错误的位置，自动以错误位置为中心提取上下文，发起专项复核与修正。
5.  支持多轮复核，若复核后仍存在争议的修改点，标记为异常，输出详细的上下文与修改内容，交由人工复核。

### 4.7 多层级校验与错误诊断模块
1.  **顺序校验**：每次处理完一卷小说，逐字符遍历文本，检测所有「和」的出现顺序，若出现连续两个相同的符号（如连续两个「、连续两个」），立即判定为顺序错误，精准输出错误位置的绝对索引、上下文内容。当通过了顺序校验的小说，将其存放到story_v1文件夹中，下一步操作基于此文件夹中的文件进行。
2.  **智能归因与多中心修正策略**：当顺序校验算法检测到符号顺序异常（如连续出现两个「）时，程序捕获的索引仅是**发现异常的检测点**，而非绝对的**错误根源点**。例如，当检测到`「(A)...「(B)`连续出现时，错误可能源于：(1) A是多余符号；(2) A后面漏了闭符号；(3) B是多余符号；(4) B本应是闭符号（OCR误识）。
    由于不同归因对应的物理位置不同（A附近 vs B附近），单一中心点的切块可能导致大模型视野受限。**算法要求**：必须实现**多视角候选修正机制**。当捕获顺序错误时，应分别以“前一个符号A”和“当前符号B”为中心生成两个独立的上下文切块（或生成一个覆盖两者的联合切块），分别提交给大模型尝试修正，或通过逻辑判断择优提交，确保大模型拥有修复任一潜在错误的完整视野。


### 4.8 结果输出与日志归档模块
1.  修正后的文本：按原始文件命名输出到指定文件夹，严格保留UTF-16编码、Unix (LF)换行符格式。
2.  变更记录：每卷文本对应一个独立的变更记录JSON文件，存储全量修改信息。
3.  校验报告：每卷文本对应一个独立的校验报告，包含校验结果、错误详情、修改统计。
4.  运行日志：全流程记录程序运行状态、大模型调用次数、处理时长、异常信息、重试记录等，输出为log文件。
5.  所有输出文件按卷号分类存储，目录结构清晰，便于查找与复核。
